Managing Dependencies using Repl.it

In this guide I will show you the ins and out of dependency management using Repl.it. If you are not familiar with Repl.it, check out this introduction to Repl.it and then come back.

Whether you like it or not, if you are developing software, you will at some point have to work with dependencies.Dependency simply means that one component relies on another component to work.

Bringing it home, when you are building a software product, you might need to use other components that other developers (internal or external) would have built to make your product work. Integrating these components and maintaining them is what dependency management is all about.

Why are dependencies useful?

It is important to make the distinction between dependencies and dependency management in order to understand the importance of each. Dependencies are useful because they prevent us from re-inventing the wheel. By that I mean that if someone has already built that component you can use, why rebuild it?

Dependency management is important for one main reason: reliability. With dependency management, you can deliver software that can be installed and run more reliably.

Apart from reliability, dependency management also helps us to achieve the following:

  1. Performance - if the components are stale and outdated, it means that we can be missing out on new enhancements which can improve performance and possibly bring out new features.

  2. Security - vulnerabilities are always found in libraries so if you don’t update the libraries your application will be exposed.

  3. Quality Assurance - to prevent problems such as circular or conflicting dependencies.

Enough theory, let’s get our hands dirty with dependency management in Repl.it. We will be using Python as our development language for the demonstration. Dependencies can also be called libraries and the two words are interchangeable.

Dependency Management in Repl.it

In this section we are going to cover the following.

  • Using libraries that we write ourselves - this is when we reference the code that we have produced ourselves. For instance, we write code that makes a calculation based on parameters that it has been given and in future we just reference that code instead of writing it all again everytime we need it.

  • Using libraries that are in the programming languages’ standard library in this case, the Python Standard Library - most if not all programming languages come with standard libraries that contain functionality that is common across programming languages and tasks. This functionality can then be used when required without us having to write it ourselves. An example of this is the code to print output on the screen, it exists already as part of the standard library and all we have to do is use it.

  • Using libraries that are external and not available in the standard library - these are libraries that add functionality to our code but are not available in the standard library. An example of this can be your proprietary code that you use in your projects.

Using libraries that we write ourselves

For libraries that we write ourselves, we don’t need to use the Universal Package Manager which is the Repl.it package manager. This is because the source code of the library is part of our project therefore we can easily change and modify it.

Now just login to repl.it and click on + new repl, choose Python as the language and give our repl an name. I am going to call mine LocalLibrary.

This is how it looks like.

And then click on Create Repl to create our repl.

When it opens up, click on Add File on the Files and Configuration pane and give our file a name.

Let’s create a very simple library. This library will only be able to do one thing - say “Good morning”. To build this module, first create a new file called ‘greeting.py’ and then enter the following code:

def morning():
return  'Good morning'

This code creates one function: morning() which returns ‘Good morning’ which is a morning greeting. In our main.py file, enter the following code:

import greeting
print(greeting.morning())

The first line of code brings in the functionality of the greeting library into our current file i.e. the main​.py file.

The second line prints the result of calling the morning() function in the greeting library.

If you press the run button, we will get the following result: an output saying “Good morning”

Hopefully this serves as a simple demonstration of how to use libraries that we write ourselves in Repl.it.

Now we move on to libraries that are the languages standard library.

Using libraries in the Python Standard Library

The Python Standard Library includes modules that provide additional functionality that you have to explicitly import in a python script. They are not built in functions and therefore do not come automatically in a new script. To give an example, in a new python script, you can run:

print('Hello World')

And it will work without any import statement at the top because the print statement is a built in function.

If you want to use say, the time or math modules, you will have to import them explicitly.

Using the Standard Libraries via the import statement is another way you can bring in dependencies in your application. Since you would not have written them, you will need to understand what functions are provided by the library that you want to use and you can find out more information about the various packages available in Python here.

Using external Libraries

Sometimes the functionality that we require exists but in a library that is not part of the Python Standard Library. In this section I will show you how you can use such a library in repl.it.

Repl.it has a package manager called the Universal Package Manager and it is responsible for bringing in and managing the third party libraries available at the Python Package Index (Pypi) into your environment.

But without explicitly using the Universal Package Manager, you can still import the Python packages.This is called the “magic import” as the complexity of importing the package is hidden from you and all you have to do is put the import statement at the beginning of your file.

For the magic import to work, you first have to make sure that the package that you want to import exists. So just open up the Python Package Index and search for the package you want. I am going to search for the requests library as I want to make an HTTP request.

Press return and you should get a list of the request libraries available. Just make sure the one that says Python HTTP for humans is available.

As you can from the above, we have requests version 2.24.0 available and this means that we can use it as a magic import. So create a new repl and call it MagicImport.

Type in the following code:

import requests
print(requests.__version__)

This code imports the requests library and then prints the version of the library that we got.

Click on run to execute the code, and we should get the following output:

Notice the first problem of magic imports - you cannot guarantee the version that you will get. When we checked the requests library on PyPi, we saw that the latest version was 2.24.0, but when we run it using the magic import, we are getting version 2.23.0.

If you run it again, we now get the current version i.e. 2.24.0.

Not being able to guarantee the version you are using is problematic because you need to trust the specific code and version you are using and not just the general library or framework it belongs to. Also updates can deprecate code that you are using or can be backward incompatible which will then break your product. So usually you are better off with specifying the version or at least the minimum version of a library that works with your product.

Specifying Library Versions

To pin versions in Repl.it, you need to provide a Spec file. Pinning versions refers to the practice of making explicit the version of the libraries your application is depending on. So for instance, if you are using requests, you can then specify the version number. This is why we need a spec file. There are different spec files for different languages for example if your project is in Javascript, you will need to provide a package.json file. Since this guide is in Python, you will need to create a pyproject.toml file.

Pyproject.toml is based on PEP 518 which seeks to Specify the Minimum Build System Requirements for Python Projects. So you basically add the build tools read libraries required for your project. You can also specify the build tool you want to use meaning that you are not constrained to using setuptools. Repl.it takes advantage of this and can use poetry project manager as the build backend.

Here is a bare minimum pyproject.toml file and a line by line explanation.

Line 1 [tool.poetry] specifies informational metadata about our package. It includes things such as the package name, version, description, author details, licences etc. Most of the config values here are optional unless you’re planning on publishing your project as an official PyPi package.

Line 2-6 is the meta-data that I have just mentioned above.

Line 8 [tool.poetry.dependencies] this is where we define dependencies our application absolutely must download to run. You may specify specific version numbers for required packages by using the format Package_name = “Version” as you see on line 9. If you want to use the latest version, you just need to set the version to an asterisk "*“. You can also specify the minimum version of a package that you are targeting but using the caret symbol followed by the version like so “^Version”. From line 9 you can deduce that the minimum version of Python required is version 3.8 and that a user running say Python 3.7 will not be able to run our project. It also means that an update is allowed if the new version number does not modify the left-most non-zero digit in the major, minor, patch grouping. In this case, Python can be updated to any version greater than 3.8 but less than 4.0 for example it can be updated to version 3.9.

Line 11 [tool.poetry.dev-dependencies] dev dependencies are not required to run the app and won’t be downloaded by default when the app is built. They are essentially packages that contributing developers should download as they iterate on the project. On Line 12 , we put down pytest = "*" which will get the latest version of pytest. Pytest is a package that will help you to write tests for your code.

Line 14 [build-system] this section is mandatory and it specifies the build tool that you are using to manage your python project. Line 15 specifies that we have to have poetry version 0.12 and above and the build-backend points to the poetry api.

If you are interested in the other config values, you can find the full list of the config values here.

If this is too much, don’t worry. Repl.it will create for us the pyproject.toml file, but it’s good to understand it. Let’s now go on and create our first pyproject.toml file with the requests library. Create a new repl and call it LibraryVersions.

Once it’s done setting up, click on the Packages icon.

It will open up a package search box:

Where you can type in and search for the package you want. We want requests so just search for it.

Select the one that says Python HTTP for Humans. And then click on the + sign.

And you can monitor the progress on the Output pane.

Now if you click on the File icon to go back to Files, you will see that apart from the main.py file, Repl.it has created for us two additional files namely: poetry.lock and pyproject.toml.

The poetry.lock file file includes all of the installed packages, locked to a specific version including the hashes so as to maintain the project integrity.

If you open up the pyproject.toml, you will see a structure similar to what we discussed above.

And as you can see, the python version is pinned at 3.8 and the requests version at 2.24.0.

If I want a specific version, this is where I would specify it and in essence pin the version. Let’s change the requests version to 2.20.0 and verify this. So update line 9 from requests = "^2.24.0" to requests = "2.20.0". Please note that if you do not remove the caret (^), it will give you back version 2.24.0.

Go to the main.py file and put in the previous code we used to check the version number of the requests i.e. put in the following code:

import requests
print(requests.__version__)

The output is that 2.20.0 will be printed out.
